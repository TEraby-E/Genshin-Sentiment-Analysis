# 分析报告：原神 B 站玩家生态与舆情分析

## 1. 业务背景与问题定义

游戏内容运营和社区团队需要持续回答三类问题：

1. **内容生态**：玩家在第三方平台（这里是 B 站）关注什么内容？哪类内容更容易被认同（点赞）、被收藏（长期价值）？
2. **舆情监控**：玩家负面情绪是否存在与版本更新相关的规律性波动？能否在波动达到运营需要介入的程度之前就识别出来？
3. **内容预测**：发布技巧（时长、标题、发布时段）和内容本身，哪个对视频能否成为爆款更重要？这决定了内容团队该把精力投在"运营技巧"还是"内容质量"上。

本项目用 Kaggle 上的原神 B 站视频/评论/官方帖子公开数据集，端到端回答上述三个问题，并对方法本身的局限性给出量化说明（而不是只给结论）。

## 2. 数据与数据质量

| 数据 | 规模 | 用途 |
| --- | --- | --- |
| 视频元数据 + 主题聚类 | 37,802 条 | 内容生态分析、爆款预测 |
| 评论 + 情感/主题聚类 | 40.8 万条 | 舆情趋势、方面级情感分析 |
| 官方帖子 | 1,607 条 | 提供评论的时间锚点（评论本身缺乏可靠时间戳，借助所属帖子的发布时间近似） |

**数据质量问题与处理**：

- 约 35% 的视频被聚类为"混合内容"（Topic_Cluster=7），人工抽样确认这是泛娱乐/无关内容混入，分析中明确剔除或单独标注，避免稀释核心内容的统计结果。
- 播放量呈极端长尾分布：头部 1% 的视频贡献了相当比例的总播放（具体数字见运行输出 `top1pct_view_share`），意味着用均值衡量"典型表现"会被极少数爆款严重扭曲——这是本项目全篇统一使用中位数而非均值的原因。
- `data_loader.py` 加载阶段加入三类契约校验（`validate.py`）：必需列是否存在、数据是否为空、日期解析成功率是否达标。这类校验的目的不是处理已知的脏数据，而是当数据源格式变化时**尽早报错**，而不是让分析悄悄产出错误结论。

## 3. 方法论

### 3.1 玩家生态：用中位数而非均值衡量主题热度

长尾分布下均值会被头部内容主导，中位数更能反映"一个该主题下的典型视频能获得多少播放/点赞"。这是面向运营决策（"该主题值不值得持续产出"）更合适的口径。

### 3.2 舆情监控：从"经验倍数阈值"到"统计显著性 + 自适应基线"

原始方法是 `负面占比 > 1.5 × 全期中位数` 触发预警。这个口径有两个问题：

1. **没有统计依据**：1.5 倍是经验数字，不能回答"这个月的上升有多大概率只是随机波动"。
2. **基线是静态的**：用全期一个中位数衡量所有月份，如果舆情基线本身随版本迭代缓慢上移或下移，固定阈值会逐渐失准（前期误报或后期漏报）。

补充了两种方法（`stats_tests.py`）：

- **双比例 z 检验**（`peak_vs_baseline_significance`）：把"峰值月 vs 基线月"看作两个独立比例的比较，计算 z 统计量和 p 值，直接回答"差异是否显著"，而不是停留在"看起来更高"。
- **滚动窗口 z-score**（`rolling_zscore_alert`）：用该月之前 N 个月（不含当月自身，避免异常值污染自己的基线）的均值和标准差，计算当月相对近期波动的偏离程度。这个口径能跟随基线漂移，提供比固定倍数更稳健的第二种预警信号。

两种口径在实际运行中相互印证（若结果分歧，通常意味着样本量不足或基线本身在剧烈变化，需要人工介入判断，而不是直接采信某一个数字）。

### 3.3 方面级情感分析：量化"细粒度分析值不值得做"

数据集自带的评论聚类是粗粒度主题簇（如"社区争议"），无法区分玩家具体在吐槽剧情、抽卡还是活动设计——这对运营来说"知道有负面情绪"和"知道负面情绪具体指向什么"是完全不同的可操作性。

`aspect_sentiment.py` 用一层可解释的关键词规则（剧情/抽卡/活动/数值与机制/运营/社交与同人六个方面），统计：

- 各方面的提及量与对应负面占比（`aspect_breakdown`），验证不同方面之间负面情绪是否有显著差异；
- **关键词覆盖率**（`agreement_with_cluster`）：多少比例的评论命中了至少一个方面关键词。这个数字本身就是判断是否需要升级到 LLM 语义分类的依据——覆盖率低，说明评论以口语化表达、表情符号为主，关键词规则的天花板有限，升级到语义级分类的边际收益更高；覆盖率高，说明关键词规则已经够用，没必要为了"看起来更高级"而引入 LLM 调用成本。

这是本项目从"做了什么"升级到"为什么这么做、什么时候该升级"的关键一环。`classify_with_llm()` 预留了接口但未实现具体请求逻辑——这是有意为之：仓库默认离线可跑，不应该因为缺少 API key 而跑不起来；需要时通过 `uv sync --extra llm` 显式启用。

### 3.4 爆款预测：用 AUC 而非准确率，因为类别不均衡

爆款定义为播放量前 20% 分位（`HIT_QUANTILE`），是天然不均衡的二分类问题，准确率会被多数类（非爆款）主导而失真，因此用 AUC 衡量模型区分能力。

特征只用元数据（时长、发布时段、星期、标题长度、视频类型），刻意不包含标题/封面的语义特征。AUC ≈ 0.60（略高于随机但远未达到强预测力）本身就是结论：**元数据层面的发布技巧对能否成为爆款的解释力有限**，爆款更可能由内容本身的质量决定——这是一个"模型效果不好"反而支撑了业务结论的例子，报告里需要明确说明，而不是把低 AUC 当作模型失败略过不提。

## 4. 关键发现

> 具体数字以 `uv run genshin-analyze` 的实时输出为准（依赖完整数据集），以下是基于既往运行的代表性结论：

1. 角色专题与二创内容在中位播放与点赞表现上显著领先于其他主题，是社区内容生态的核心。
2. 负面舆情存在与版本节奏相关的月度峰值（峰值 21.5% vs 基线 ~10.7%），双比例 z 检验确认该峰值相对基线月份的差异具有统计显著性（p < 0.05）。
3. 关键词方面分析显示，"数值与机制"和"运营"两个方面的负面占比明显高于"剧情"和"社交与同人"，说明负面情绪更多指向可运营调整的范畴（数值平衡、客服响应），而非内容本身的接受度问题。
4. 爆款预测 AUC ≈ 0.60，特征重要性中内容相关特征（如标题长度、视频时长）的重要性高于纯发布技巧特征（发布时段、星期），支持"内容质量主导"的结论。

## 5. 局限性

- **时间锚点近似**：评论本身没有可靠时间戳，用所属帖子的发布时间近似代表评论产生时间，存在滞后误差（评论可能在帖子发布后持续产生数周）。
- **关键词规则的天花板**：方面级情感分析依赖人工列举的关键词列表，无法识别隐喻、反讽或新出现的玩家黑话；`agreement_with_cluster` 的覆盖率指标已经量化了这一局限的大小。
- **爆款预测特征有限**：未引入标题/封面/正文的语义特征，AUC 上限因此被结构性限制，不能据此判定"内容质量"本身的可量化贡献，只能反证"现有元数据特征解释力不足"。
- **样本量不均衡**：部分月份评论量低于 `MIN_MONTHLY_COMMENTS` 阈值被直接剔除，意味着早期或晚期数据覆盖可能不连续，趋势图的两端可信度低于中段。

## 6. 后续方向

- 接入 LLM 做语义级方面分类，与关键词基线对比，用一致性指标（如 Cohen's Kappa）量化升级的实际收益，而不是直觉判断"LLM 应该更准"。
- 引入标题/封面的语义特征，重新评估爆款预测模型，检验"内容质量主导"结论在更丰富特征下是否依然成立。
- 把舆情监控做成事件驱动的更细粒度（周/日）口径，结合官方帖子发布时间，识别具体哪类公告/活动最容易引发负面情绪峰值。
